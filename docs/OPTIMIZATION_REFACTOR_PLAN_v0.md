# OriginLab MCP 完整优化与重构方案（v0 初版）

> 文档目的：给出可执行的重构蓝图，并明确 **v0.1 初版存在重大功能欠缺**，避免误用为生产方案。

## 0. 当前版本状态说明（必须阅读）

当前版本为 `v0.1.0`，定位为早期可用原型：

- 功能覆盖广，但实现集中在少数大文件中，维护成本高。
- 错误处理和参数校验风格尚未完全统一。
- 部分流程具备“可用性”但缺少“生产级稳定性保证”。
- 测试以基础能力为主，缺乏系统级回归矩阵。

**结论：该版本存在重大功能欠缺，不建议用于生产环境。**

---

## 1. 重构目标

1. 结构可维护：拆分超大工具模块，降低改动耦合。
2. 行为可预测：统一错误模型与校验入口。
3. 交付可回归：建立按工具域划分的测试矩阵。
4. 文档可落地：让贡献者按固定流程迭代。

---

## 2. 分阶段执行计划

### Phase A（稳定性优先，1~2 周）

- 补齐高风险参数校验（导出宽度、索引范围、非法空值）。
- 统一 `error_response` 输出结构和错误枚举映射。
- 将 CSV 导出切换为标准 `csv` writer，避免转义问题。

**验收标准**
- 错误结构在主要工具中一致。
- 导出类工具包含边界测试。

### Phase B（结构重构，2~4 周）

- 拆分 `tools/customize.py` 为多个子模块（axis/style/label 等）。
- 拆分 `tools/data.py` 为 import/read/write/manage 子模块。
- `register_*_tools` 保持兼容，仅做装配层。

**验收标准**
- 单文件行数显著下降。
- 对外工具名与调用方式保持不变。

### Phase C（质量体系，1~2 周）

- 增加 `tests/test_*_tools.py` 按域分层测试。
- 引入参数化测试覆盖常见非法输入。
- 补充 Origin 不可用场景的容错测试（mock/stub）。

**验收标准**
- 关键工具拥有 success/fail/edge 三类用例。
- 回归失败能够准确定位到工具域。

### Phase D（体验优化，持续）

- 统一中英文日志与错误信息策略。
- 为常见工作流提供“建议下一步”模板。
- 增加高级工具（LabTalk）风险提示等级。

## 4. 风险与回滚策略

- 风险：重构中工具注册遗漏导致 MCP 暴露能力减少。
  - 处理：增加注册清单快照测试。
- 风险：错误语义变更影响调用方。
  - 处理：保留旧字段并在文档标注 deprecate。
- 风险：拆分导致循环依赖。
  - 处理：将共享逻辑收敛到 `utils/helpers.py` 与 `utils/validators.py`。

---

## 5. 版本承诺

- `v0.1.x`：以修复和稳定性为主，不承诺完整兼容。
- `v0.2.0`：目标完成结构重构和测试矩阵升级。
- `v1.0.0`：在完成生产级回归与文档闭环后再考虑。
